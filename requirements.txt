# Telecom-T2C-Trainer — pinned dependencies
#
# Anchored to the reference notebook's already-proven-working floor versions:
#   transformers>=4.51.0, trl>=0.15.0, peft>=0.14.0, bitsandbytes>=0.45.0, accelerate>=1.2.0
#
# transformers / tokenizers / huggingface_hub are deliberately FLOOR-ONLY, not
# pinned to an exact version. google/gemma-4-12B-it was released after this
# project's authoring, so the true minimum transformers version that
# recognizes its architecture is not knowable ahead of time — pinning a
# guessed exact number risks being wrong in either direction (too old ->
# unrecognized model_type; a fabricated "looks right" number -> may not
# actually exist). src/model.py's load_base_model() has a runtime capability
# check with an actionable error message ("pip install -U transformers") as
# the real safety net here, not this pin. At install time, prefer the newest
# available release of these three packages.
transformers>=4.51.0
tokenizers>=0.21.0
huggingface_hub>=0.26.0

# peft and trl are ALSO floor-only, not exact-pinned, as a general defensive
# practice: since transformers is left free to resolve much newer than this
# project's floor, an exact-pinned old peft/trl release could in principle
# predate an internal transformers API it depends on. (One real failure that
# LOOKED like this — "Could not import module 'BloomPreTrainedModel'" while
# importing peft — turned out to be unrelated: see the torch note below. Kept
# floor-only regardless, since the underlying risk of peft/trl lagging a
# much-newer transformers is still real even if it wasn't this incident's
# cause.)
peft>=0.14.0
trl>=0.15.0

# torch is intentionally NOT pinned/listed here at all — this is the fix for
# a confirmed, reproduced incident: pinning torch (even loosely) makes pip
# reinstall it in Colab with whatever CUDA-toolkit-tagged wheel satisfies the
# constraint, which does NOT necessarily match the CUDA build of Colab's
# pre-installed torchaudio/torchvision. Symptom observed: `import peft`
# failing with "Could not import module 'BloomPreTrainedModel'" — the real
# underlying exception (visible via the notebook's Install-section traceback)
# was `RuntimeError: Detected that PyTorch and TorchAudio were compiled with
# different CUDA versions (PyTorch 12.4 vs TorchAudio 12.1)`, raised deep in
# a transformers audio-loss import chain that peft's own import happens to
# pull in — nothing to do with peft/trl versions at all. Colab's pre-installed
# torch is already correctly matched to its own CUDA driver and to
# torchaudio/torchvision; leaving it alone (never pip-installing/upgrading it)
# is the safest choice for this project, which only ever targets Colab.
# transformers/peft/trl/accelerate/bitsandbytes all declare torch as a loose
# transitive dependency, so `pip install -r requirements.txt` is satisfied by
# whatever torch is already present without us needing to name it explicitly.

# Everything below IS pinned to a concrete, mutually-compatible point release
# — these have no equivalent tight coupling to transformers' internal
# per-architecture class exports or to torch's own CUDA build, so
# exact-pinning them is safe.
datasets==3.2.0
accelerate==1.2.1
bitsandbytes==0.45.1
sentencepiece==0.2.0
PyYAML==6.0.2
protobuf==5.29.3

# numpy is FLOOR-ONLY, not exact-pinned, for the same reason as the
# transformers trio above, plus a concrete Colab-specific failure mode:
# hosted Colab images ship numpy 2.x by default with pandas/pyarrow wheels
# already compiled against its (larger) dtype ABI. Forcing an exact numpy
# 1.x pin here downgrades numpy without also rebuilding those already-
# compiled binaries, producing "ValueError: numpy.dtype size changed, may
# indicate binary incompatibility" on the first `import pandas` (pulled in
# transitively by `datasets`). Leaving numpy unpinned-below-3 lets pip keep
# it aligned with whatever pandas/pyarrow the Colab image already has.
numpy>=1.26,<3
safetensors==0.4.5

# Experiment tracking. Gracefully no-ops via src/wandb_logger.py if unavailable/unauthenticated.
wandb==0.19.1

# Optional — used only by src/utils.get_gpu_stats() for GPU utilization %.
# Import is guarded (try/except) everywhere it's used; omit this line entirely
# if installation fails in your environment, memory stats will still work.
nvidia-ml-py==12.560.30
