# Telecom-T2C-Trainer — pinned dependencies
#
# This project is Unsloth-only (see src/model.py / README "Model backend").
# transformers==5.12.1 is exact-pinned deliberately: Gemma 4's tokenizer
# config requires transformers v5 outright (v4.x cannot load it at all —
# see tokenizer.py's patch_extra_special_tokens_list_format() docstring),
# and 5.12.1 is the specific release this project has actually been
# validated against. peft/accelerate/bitsandbytes/datasets are floor-pinned
# just above that combination rather than exact-matched to it, since they
# still need room to pick up compatibility fixes for it (see the peft note
# below) without requiring a requirements.txt edit each time.
transformers==5.12.1
tokenizers>=0.21.0
huggingface_hub>=0.26.0

peft>=0.19.1
trl>=0.15.0
accelerate>=1.8
datasets>=3.2

# Known, currently-unresolved upstream risk this pin set does NOT fully
# rule out: peft has had no working PyPI release for transformers>=4.55 in
# the past (confirmed — https://github.com/huggingface/peft/issues/2754,
# "No working peft version available in PyPI for transformers 4.55+";
# reproduced during this project's development as `ImportError: cannot
# import name 'BloomPreTrainedModel' from 'transformers'` at a standalone
# `import peft`). If that surfaces again despite the floor above, it's a
# peft/transformers compatibility gap to track upstream, not a bug in this
# project — see README Troubleshooting. It does not necessarily block the
# Unsloth-mediated LoRA path (`from unsloth import FastModel`) even when a
# standalone `import peft` fails, since Unsloth may not exercise the exact
# broken code path — which is also why the notebook's Section 2 version
# check no longer probes `peft` directly, only `unsloth`/`unsloth_zoo`.

# unsloth: custom kernels/patches for a curated set of architectures,
# confirmed to include the Gemma 4 family (unsloth/gemma-4-12b-it exists on
# the Hub), typically cutting VRAM usage substantially for QLoRA versus a
# plain transformers+bitsandbytes path. Deliberately left UNPINNED (not
# even floor-only) — it needs to resolve a version actually compatible with
# whatever transformers/peft/trl/torch/CUDA Colab has *today*, and
# Unsloth's own installer explicitly recommends auto-detection
# (`--torch-backend=auto`) over manual pinning for this reason.
#
# unsloth_zoo is listed explicitly too, not left to resolve transitively:
# unsloth's own exec()-based patching of transformers internals periodically
# breaks when transformers renames something internal (confirmed pattern —
# see https://github.com/unslothai/unsloth/issues/3415), and the
# maintainers' fix for that class of bug has historically landed as
# companion PRs across BOTH unsloth and unsloth_zoo. Leaving unsloth_zoo
# unlisted would let it resolve independently and possibly miss the paired
# fix even when `unsloth` itself is current.
unsloth_zoo
unsloth

# torch is intentionally NOT pinned/listed here at all — confirmed,
# reproduced incident: pinning torch (even loosely) makes pip reinstall it
# in Colab with whatever CUDA-toolkit-tagged wheel satisfies the
# constraint, which does NOT necessarily match the CUDA build of Colab's
# pre-installed torchaudio/torchvision, causing native-extension crashes
# deep in unrelated import chains. Colab's pre-installed torch is already
# correctly matched to its own CUDA driver and to torchaudio/torchvision;
# leaving it alone (never pip-installing/upgrading it) is the safest choice
# for this project, which only ever targets Colab. Every package above
# declares torch as a loose transitive dependency, so `pip install -r
# requirements.txt` is satisfied by whatever torch is already present
# without us needing to name it explicitly.

# Everything below IS pinned/floored to a concrete release — no equivalent
# tight coupling to transformers'/peft's internal APIs or to torch's own
# CUDA build, so this is safe.
bitsandbytes>=0.46.1
sentencepiece==0.2.0
PyYAML==6.0.2
protobuf==5.29.3

# numpy is intentionally NOT listed here at all — same reasoning as torch
# above, confirmed by a second, distinct incident: even a floor-only numpy
# pin still gets touched by `pip install --upgrade` (required so other
# floor-pin bumps in this file actually apply on re-installs), and that
# in-place upgrade left numpy in a broken hybrid state on at least one
# Colab image (numpy's own pure-Python layer and compiled C-extension layer
# out of sync). Any package whose Colab-preinstalled build is load-bearing
# for binary compatibility needs to be left out of requirements.txt
# entirely, not just loosely pinned. src/utils.py's only numpy usage (in
# set_seed) is already try/except-guarded, so this project has no hard
# dependency on numpy being present at all — whatever datasets/torch/
# transformers pull in transitively is sufficient.
safetensors==0.4.5

# Experiment tracking. Gracefully no-ops via src/wandb_logger.py if unavailable/unauthenticated.
wandb==0.28.1

# Optional — used only by src/utils.get_gpu_stats() for GPU utilization %.
